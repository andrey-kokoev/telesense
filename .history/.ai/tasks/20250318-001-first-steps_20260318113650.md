# Task: Get `telesense` to first verified 1:1 Cloudflare Realtime call

## Objective

Inside repo `telesense`, create the **smallest possible, verification-first foundation** for a raw Cloudflare Realtime 1:1 video call path.

This task is **not** to build a polished app.  
This task is to move from **zero** to a **verified path to “1”**:

- one browser tab publishes
- second browser tab joins
- both sides complete the required negotiation
- audio/video can flow
- all Cloudflare request/response shapes used in code are grounded in captured or verified payloads

---

## Non-goals

Do **not** add any of the following unless absolutely required by the verified protocol:

- Durable Objects
- Web Workers
- multiparty room abstractions
- auth system beyond secure backend secret handling
- chat
- presence UI
- screen share
- reconnect sophistication
- UI polish
- production deployment hardening

## Phase 3 — Minimal implementation scaffold

Once the wire contract is good enough to support a minimal spike, create the thinnest implementation.

### 6. Backend requirements

Implement a Worker in `src/worker/index.ts` with only these app-facing routes:

- `POST /api/calls/:callId/session`
- `POST /api/calls/:callId/publish-offer`
- `POST /api/calls/:callId/complete-subscribe`
- `GET /api/calls/:callId/discover-remote-tracks`
- `POST /api/calls/:callId/leave`

Guidelines:

- single file is acceptable for initial spike
- keep state in in-memory `Map` only for dev
- clearly label in code that in-memory state is invalid across isolates/restarts/deploys
- keep Cloudflare secret server-only
- fail hard on unexpected upstream payload shape
- no silent fallbacks once a shape is verified

### 7. Backend code standards

Implement these behaviors:

- strict validation for all browser-facing JSON bodies
- explicit 400/404/502 responses
- upstream Cloudflare failures logged with route and status
- raw response body logged only behind a debug flag
- no `any` unless absolutely unavoidable during protocol discovery
- use provisional types only where still unresolved
- keep comments marking remaining wire-format blockers

### 8. Client requirements

Implement `src/client/main.ts` with the absolute minimum:

- capture local camera/mic
- create `RTCPeerConnection`
- publish local offer through backend
- set returned answer as remote description
- mount local and remote video elements
- poll or otherwise follow verified mechanism for discovering remote tracks / completing remote subscribe
- no framework needed unless repo already uses one and it is cheaper to reuse than bypass

### 9. HTML requirements

Create `public/index.html` with only:

- local video element
- remote video element
- minimal status text area
- script tag/module for client entry

## Implementation constraints

### 12. Keep architecture minimal

Do not introduce:

- Durable Objects
- database
- websocket signaling layer
- abstractions for future multiparty use
- room model beyond simple `callId`
- elaborate service classes

### 13. Naming constraints

Use these route names exactly unless verified protocol demands otherwise:

- `/session`
- `/publish-offer`
- `/complete-subscribe`
- `/discover-remote-tracks`
- `/leave`

### 14. State constraints

Use app-level rendezvous state like:

```ts
Map<
  callId,
  Map<
    internalSessionId,
    {
      cfSessionId: string;
      publishedTrackIds: Set<string>;
    }
  >
>;
```

Important:

- store only Cloudflare-assigned track IDs for remote discovery
- do not use browser `MediaStreamTrack.id` as remote subscription identifier
